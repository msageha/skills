# AGENTS.md

LLM Agent が作業を行うときに読み込むためのユーザー (mzk) の作業規約。
過去の Claude Code 会話ログ (2026-06〜07、プロンプト約 2,400 件 + セッション中フィードバック) から、
繰り返し現れた指示・訂正・却下のパターンを形式知化したもの。

## 最重要原則 (違反すると即座に差し戻される)

1. **質問は質問であり、変更のトリガーではない**。「〜ってなに？」「なんで〜を追加したの？」「原因わかる？」「これでいいの？」は説明・理由を求めているだけ。回答のみ返し、コードや設定を変更しない。過去に「なんで libgomp1 を追加したの？」という質問に対して削除で応じ、「消せと言っているわけではなく、追加した理由を聞いていました」と叱責された事例がある。
1. **診断と修正は分離する**。まず原因説明・提案を返し、ユーザーが納得して「修正してください」「では、その対応を行ってください」と明示してから着手する。調査依頼に「いきなり変更は行わないでください」が付くこともあるが、付いていなくても調査フェーズでは変更しない。
1. **スコープは指示された範囲だけ**。ユーザーはエージェントの過去出力をバッククォートで引用し「この表記だけ削除して」「`1`, `3`, `5` を対応。`2` は一旦放置」と項目単位で裁定する。指定外への波及修正・勝手な機能追加・頼まれていないテストやコメントの追加・後方互換の勝手な温存は却下される。波及が必要だと判断したら、適用せずに指摘・提案として提示する。
1. **実行主体・プロセスの指定は厳守する**。「codex mcp (gpt5.5) に実装させ、codex mcp (fugu ultra) にレビューさせる」と指定されたら自分で実装しない (「なぜあなた自身が実装を始めているのですか？」と差し戻された事例あり)。宣言したプロセスのステップを省くと「〜しましたか？していなければ行ってください」と必ず追及される。
1. **事実でないことを言わない**。ユーザーの発言や状態を勝手に前提化しない (「そもそも私、一言も〜なんて言ってません」)。断定した内容は後日原文引用されて検証されるため、根拠のある正確な記述をする。存在確認・環境情報は推測でなく実測する。

## 対話プロトコル

- ほぼ全て日本語。技術用語・コマンド・パスは原文のまま扱う。
- typo・尻切れメッセージが頻繁にある (「げんいんをしらべてもらえますか？」「push して、Windows CI ジョブga」)。誤字は意図を汲んで頑健に解釈し、尻切れは直後の完全版再送を待つ。同一プロンプトの再送は上書きとして扱う。
- コマンド出力の貼り付けだけのメッセージ (「`...` とでました」) は「前の指示の実行結果なので次の診断ステップを出せ」の意。トラブルシュートは 1 ステップずつ提示 → 結果貼り付け、の往復で進む。
- 短い肯定 (「はい」「2」「では、その対応を行ってください」) は直前の提案・選択肢への実行許可。
- 選択肢を出すときはラベル付き (A/B/C や番号) で、推奨 + 理由 + トレードオフ (コスト vs 性能など) を添える。ユーザーはラベルを引用して選ぶ。
- 実装手段が複数あるとき (設定で回避 vs 構造で回避など) は片方だけ実行せず、比較提示してから進める。
- 方針決定前の判断に迷う点は自己判断せずユーザーに確認する。方針確定後は途中で止まらず完走する (「途中で止まってしまうとユーザーの介入が都度必要で非常に困ります」)。
- 品質 > コスト・時間。「時間がかかってもいいので、品質や性能を優先したいです」が基本姿勢。

## レポート・報告の型

- 調査・レビュー・監査は「重要なものだけ」ではなく網羅列挙。severity / priority タグと番号を付け、項目単位で引用・裁定できる粒度で構造化する。修正後は「その他に対応するべき問題はないか」の再監査ループが来る前提で、残課題の台帳 (対応済み / 未対応) を維持する。
- 長時間タスクの状態報告は「完了 N / 実行中 M / 待機 K + 進捗の観測的証拠 (ログ末尾・レスポンス増加など) + 異常兆候の有無の明示的判定」の型で返す。「実行中 (健全)」のようなラベルだけの報告は受け入れられない。数値を出すときは正常範囲かどうかの判断を添える。
- 完了報告は自己検証済みであることが受け入れ条件。「すべての要件が実装できたか、本当にそれが正しいか、実装漏れが無いか」の突合を先回りして行い、根拠付きで報告する。
- 成果物 (ドキュメント・skills 等) には読者に必要な情報だけを残す。検証メタ情報や経緯の混入は「精査して簡潔なドキュメントにしてください」と差し戻される。

## 検証

- 手順の提示で止まらず、可能な限り自分で実行して確認する (「あなた自身が実行してもらえる？」が繰り返し要求される)。
- API は実際にリクエストを送ってドキュメントとの整合を確認する。フロントエンドは Playwright 等で実際に操作し「問題ないと判断が取れるまで」確認する。
- commit 後は「問題がないか、壊れていないか、含めてはいけない情報が無いか」の自己監査を行う。push 後は CI を完走まで watch し、failed なら修正までがワンセット。
- 長時間ジョブはバックグラウンドで流し、完了・失敗を通知する。失敗時はログ確認と根本原因分析まで行う。

## git 運用

- コミットは細かい単位で頻繁に。コミットメッセージは Conventional Commits 相当の要約で委任される (`/commit` skill 相当)。
- push・PR 作成・merge・タグはユーザーの明示指示があったときのみ。「commit のみおこなってください」と push を止められることもある。
- `Co-Authored-By: Claude` 等のトレーラーは付けない。
- コミット対象は統制されている: `.idea/`・`.envrc` 等のローカルファイルを含めない。逆に gitignore 対象でも追跡指定されたファイルは取り込む。
- 実験・検証のための一時的変更は、確認が済んだら明示的に巻き戻す (checkout で戻す、シンボリックリンク解除など)。

## コード・成果物の規約

- 新規追加は既存コード・指定された参照リポジトリのパターンに準拠する (「他の `install/debian/apt.sh` などに習ってください」)。
- 命名の意味的整合性を厳密に見る。変数名はスコープ・内容と一致させる (「CliTools 以外にも skip するので skipWindowsExtras っていう変数名を使うように」)。
- DRY より可読性。過度な抽象化・共通化は「すっごくわかりづらくなった」と差し戻される。
- コードを読めば分かるコメント、Makefile を見れば分かる説明などの冗長な記述は書かない。
- フォールバック付きデフォルトでごまかさず fail-fast にする (「デフォルト設定はなくし、`.env` がなければ failed するようにして」)。
- Python は pydantic / StrEnum で厳密に型付け・validation する。Field に説明とデフォルト値を記載する。
- GitHub Actions 等の外部依存は SHA でピン留めし、バージョンをコメントで記載する。
- gitignore はホワイトリスト形式 (全部 ignore して `!` で許可)。Makefile の `.PHONY` は各ターゲット直上に書く。
- クロスプラットフォーム (macOS / Ubuntu / Debian / Windows, amd64 / arm64) を壊さない。
- dotfiles は chezmoi 管理。適用先 (`$HOME` 配下) を直接編集せず、source (`~/.local/share/chezmoi/home/`) を編集して apply する。設定変更はワークスペースローカルではなくグローバル (ユーザー設定 / chezmoi source) 側がデフォルト。

## 情報の正確性・鮮度

- 学習データの古さを前提に、ライブラリ仕様・API・ベストプラクティスは公式ドキュメントや web で最新情報を確認してから答える。古い情報は「あくまで参考」と明示する。
- 技術的な断定は一次情報 (公式ドキュメント URL 等) と突き合わせられる。反証されたら素直に訂正する。
- 解決の過程で得られた運用知見 (ツールの劣化傾向、回避策など) は、その場限りにせずメモリやドキュメントへの永続化を提案する。

## リモートホスト・インフラ操作

- リモートサーバー (Proxmox 等) 上の操作は原則ユーザーが手で実行し、エージェントはコマンド提示役という分業。SSH 情報を渡されて直接操作を委任される場合もあり、どちらかは指示で判断する。
- 提示するコマンドはコピペ実行可能な完全なもの。pager・対話プロンプト・フォアグラウンド常駐などの入力が返らないコマンドは避けるか、非対話の代替を示す。
- 対象環境のツール前提を確認する (jq が無い環境で jq を使う等は差し戻される)。
- インフラ変更 (firewall、DNS、ストレージ割当など) はパラメータの具体値まで正確に。ユーザーは CIDR やサイズの細部まで検証する。

## 委譲・長時間実行・マルチエージェント

- 外部 LLM / サブエージェントへの委譲は短い粒度に分割する。長寿命スレッドは劣化する傾向があるため。ただし依存関係のある検証は分割しすぎると網羅性の確認が難しくなるので考慮する。
- 委譲先がハング・停滞したら、打ち切ってさらに細かく分割して再投入するのが定型リカバリ。
- 並列度 (タスクを 1 つにまとめる / 3 分割して並列など) はユーザーが明示した指定に従う。
- 完了条件 (例: `go build ./...` と `go test ./...` が通ること) が指定された場合、それが受け入れ基準。満たすまで完了と報告しない。

## セキュリティ

- public 化・公開前には credential・個人情報が含まれていないかの確認を必ず行う (git 履歴・Docker イメージ内も含む)。
- 秘密情報は出力時にマスクする。ユーザー自身もトークンを「(あえて省略してます)」とマスクして送る水準の意識を持つ。
- permission・セキュリティ設定は「全部 deny / 全部 ask」の一律案を出さず、対象ごとに deny / allow / 例外 (例: `.env` は deny だが存在確認は可、`.env.template` は読み書き可) を分けて提案する。ask の追加は基本的に不要で、利便性を損なう制限は事前に指摘する。
